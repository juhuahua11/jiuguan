# 记忆提取速度修复方案

## 问题

`memory-proxy-plugin` 的提取管线用 `__PROCESSING__` sentinel 防止并发提取，但高频聊天时 sentinel 一直 fresh → 后续请求全部跳过 → 提取永远追不上消息增长速度 → 记忆库中缺失最近对话的事实/事件/关系。

### 当前流程

```
请求1 → scheduleExtraction → 设 __PROCESSING__ sentinel → 异步提取
请求2 → 检查 sentinel (fresh) → × 跳过
请求3 → 检查 sentinel (fresh) → × 跳过
...
提取1完成 → 清除 sentinel → 指纹推进
请求N → 差量提取（含请求2/3的消息）
```

**问题**：提取跑 2-5 分钟，期间可能产生 5-10 轮对话，全部被跳过。虽然指纹机制保证不永久丢失（跳过消息在下次差分时重试），但：
- 消息量持续增长 → 差分越来越大 → 提取更慢 → 恶性循环
- 最近轮次的记忆不回库 → 模型出现"失忆"

---

## 方案对比

### 方案 A：Pending 标记自排空 ⭐ 推荐

**思路**：跳过时不丢，提取完成后自动追一轮

```
请求1 → 提取开始 → sentinel + pending=false
请求2 → sentinel fresh → 设 pending=true，跳过
请求3 → sentinel fresh → pending 已为 true，跳过
提取1完成 → 清除sentinel → 检查 pending=true → 自动触发提取2（覆盖请求2+3）
提取2完成 → 清除sentinel → 检查 pending=false → 结束
```

**改动**：
1. `sessions` 表加 `extraction_pending INTEGER DEFAULT 0`
2. `session-store.ts` 加 `setExtractionPending(sessionId, pending)` / `getExtractionPending(sessionId)`
3. `memory-handler.ts` 的 `scheduleExtraction`：sentinel fresh 时设 pending=true；提取完成后检查 pending → 递归调用

**优点**：
- 自排空，不丢数据
- 不浪费 LLM 调用（每轮覆盖更多消息，效率更高）
- 改动量小（~50 行）

**缺点**：
- 极端高频下会连续 chaining（通过限制 maxChain=3 解决）

---

### 方案 B：去 Sentinel，用指纹 CAS

**思路**：不阻塞并发，提取完成时 CAS 更新指纹（指纹未变才写入）

```
请求1 → 提取（快照: 指纹A）
请求2 → 提取（快照: 指纹B）
提取2先完成 → CAS(预期B→B') → 成功（新指纹匹配）→ 提取2覆盖请求1+2
提取1后完成 → CAS(预期A→A') → 失败（指纹已变）→ 丢弃
```

**改动**：
1. 移除 sentinel 相关逻辑
2. `updateSessionExtractionProgress` 改为 CAS：`UPDATE ... SET last_fingerprint=? WHERE session_id=? AND last_fingerprint=?`
3. 提取前保存预期指纹，完成后提交

**优点**：
- 完全不阻塞
- 永远是最新提取胜出
- 天然去重（旧提取自动丢弃）

**缺点**：
- 可能浪费 LLM 调用（旧提取白跑一趟）
- 两个提取并发写 DB，虽在事务内但逻辑复杂
- 涉及改动较大（~150 行）

---

### 方案 C：提取触发从"每轮"改为"空闲时"

**思路**：不在每轮对话后触发，而是在用户暂停聊天 N 秒后才触发

```
发消息 → 重置 debounce 计时器 30s
发消息 → 重置 debounce 计时器
...停止聊天...
计时器到 30s → 触发提取（此时消息已稳定）
```

**改动**：
1. `memory-handler.ts` 中 `doExtraction()` 不立即调用，改为 `debounceExtraction()`
2. 每次请求重置计时器
3. 计时器到期 → 真正触发提取

**优点**：
- 自然避免积压（用户暂停才提取）
- LLM 调用大幅减少
- 提取时的消息状态是稳定的

**缺点**：
- 如果用户持续不断聊天 30 分钟，提取永远不会触发
- 需要额外保护（如每 20 轮强制触发一次）

---

## 🎯 推荐方案：A + C 组合

```
触发策略 (C)：
  - 每 3 轮对话 或 20 秒空闲 → 触发提取
  - 每 20 轮强制触发一次（安全兜底）

追赶机制 (A)：
  - 提取完成 → 检查 extraction_pending
  - pending=true → 自动再跑一轮（最多链式 3 次）
```

### 具体改动点

| 文件 | 改动 |
|------|------|
| `memory-proxy/src/storage/db.ts` | `sessions` 表加 `extraction_pending` 列 |
| `memory-proxy/src/storage/session-store.ts` | 加 `setExtractionPending()` / `getExtractionPending()` |
| `memory-proxy-plugin/src/server/memory-handler.ts` | debounce 触发 + pending 检查 + 递归追赶 |

### 预期效果

| 聊天频率 | 提取落后 | 记忆质量 |
|----------|----------|----------|
| 低（>1min/轮） | 0-1 轮 | 🟢 正常 |
| 中（30s/轮） | 1-3 轮 | 🟢 几乎无感知 |
| 高（10s/轮） | 3-6 轮 | 🟡 轻微延迟 |
| 极高（<5s/轮） | 受限于 maxChain=3 | 🟡 最多落后 3 批 |

---

## 实施建议

1. **先实施方案 A**（改动小、见效快、零风险）
2. 观察日志中的 `skipping extraction` 频率
3. 如仍有积压，再加方案 C 的 debounce 触发

---

## 不推荐的方案

- **加大 sentinel 超时**：治标不治本，慢提取还是慢
- **并行提取**：sql.js 单连接，并行写会死锁
- **降低提取质量**：减少 token/cut chunk → 提取变快但质量差，得不偿失
