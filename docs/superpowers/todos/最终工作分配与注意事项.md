# 最终工作分配与注意事项

> 主题：`[FIX: memory-extraction-backlog]`
>
> 本文档用于三方协作落地 memory extraction backlog 修复。当前共识来自 `共享文档.md` 中 Claude、DeepSeek、GPT 三方讨论。

---

## 一、最终执行顺序

```text
P0：Claude Code 先做 repair 请求绕过 memory side effects
↓
P1：DeepSeek-v4-pro 再做 pending / catch-up 主逻辑
↓
P1-storage + Review：GPT 做 storage/session-store 小改、review、验收、文档同步
```

注意：`P0` 和 `P1` 都会碰到 `memory-proxy-plugin/src/server/memory-handler.ts`，但改动区域不同。必须串行推进，不能三方同时改同一文件。

## 二、Claude Code 分工：P0 repair 绕过

### 负责范围

```text
memory/adapter.js
memory-proxy-plugin/src/server/memory-handler.ts
```

### 具体任务

1. 在 `memory/adapter.js` 的 repair 请求体里增加内部标记：

```js
jiuguan_internal_repair: true
```

2. 在 `memory-proxy-plugin/src/server/memory-handler.ts` 的 messages 验证之后加入 internal repair fast path。

3. fast path 必须复用主路径已有的上游请求逻辑，例如 `fetchOnce` 或同等 helper。

4. fast path 明确跳过所有 memory side effects：

```text
不创建 / 更新 session
不递增 round
不 keyword refresh
不 continuity snapshot rebuild
不 memory retrieval / memory injection
不 scheduleExtraction
不写 memory DB
```

### Claude Code 边界

```text
只改 P0 相关区域。
在 memory-handler.ts 中只动顶部 fast path。
不要改 scheduleExtraction / extraction 后半段。
不要改 storage / DB migration。
```

### 验收日志

```text
[MemoryProxy] Internal repair request: bypassing memory side effects
```

## 三、DeepSeek-v4-pro 分工：P1 pending / catch-up 主逻辑

### 负责范围

```text
memory-proxy-plugin/src/server/memory-handler.ts
```

### 具体任务

1. 把 `scheduleExtraction` 中真正的 extraction 主体抽成：

```ts
runExtractionOnce({ sessionId, requestMessages, source })
```

2. `runExtractionOnce` 必须复用完整主流程，至少包括：

```text
读取 fingerprint / last_message_count / integrity hash
integrity hash 校验
__PROCESSING__ sentinel zombie 检查
diffNewMessages
chunkMessages
markExtractionInProgress
runExtractionPipeline 循环
fingerprint 推进 / 失败恢复
clearExtractionSentinel
```

3. sentinel fresh 时改成：

```ts
setExtractionPending(sessionId, true);
pendingExtractionMessagesBySession.set(sessionId, requestMessages);
console.log('[MemoryProxy] Extraction already in-progress; marked pending catch-up');
return;
```

4. normal extraction 完成后检查 pending：

```text
如果 extraction_pending = true：
  从 pendingExtractionMessagesBySession 读取最新 requestMessages
  清 pending
  调 runExtractionOnce({ source: 'catchup' })
```

5. catch-up 不递归：

```text
catch-up 结束后不再立即检查 pending
catch-up 期间如果又有新请求，只重新 mark pending
下一次真实请求或下一轮 extraction 再处理新的 pending
```

### DeepSeek 边界

```text
不要改 adapter.js
不要改 storage / db.ts / session-store.ts
不要直接把 runExtractionPipeline 当成完整 catch-up
不要新造一条并行写库路径
不要让 catch-up 递归
```

## 四、GPT 分工：storage/session-store + review + 验收

```text
memory-proxy/src/storage/session-store.ts
```

### 具体任务

```sql
ALTER TABLE sessions ADD COLUMN extraction_pending INTEGER NOT NULL DEFAULT 0
```

2. 在 `session-store.ts` 增加：

```ts
setExtractionPending(sessionId: string, pending: boolean): void
getExtractionPending(sessionId: string): boolean
```

```ts
extraction_pending?: number | boolean
```

4. Review Claude Code 的 P0 diff：


## 五、协作注意事项

### 1. 串行推进

```text
Claude Code 完成 P0 并 push
→ 所有人 pull
→ DeepSeek 基于最新 main 做 P1
→ DeepSeek push
→ 所有人 pull
→ GPT review / storage / 验收
```

不要三方同时改 `memory-handler.ts`。

### 2. 文件边界

```text
Claude Code：adapter.js + memory-handler.ts 顶部 fast path
DeepSeek：memory-handler.ts 后半 scheduleExtraction / runExtractionOnce
GPT：storage/session-store/types + review + docs
```

### 3. 必须保护的现有功能

```text
replace_branch 流式修复不能回退
Prompt Compiler 不能被改弱
世界书预算不能被绕过
watchdog repair / deterministic fallback 不能失效
正常用户请求的 memory retrieval 不能被跳过
```

### 4. 不推荐做的事

```text
不要去掉 sentinel 改 CAS 并发 extraction
不要做 debounce idle extraction 作为第一版
不要直接缩小 extraction chunk 牺牲质量
不要把 sentinel timeout 粗暴改短
不要允许同 session 多个 extraction 并发写库
不要让 repair 请求继续进入完整 memory 链路
```

## 六、验收标准

修复完成后应该看到：

```text
[MemoryProxy] Internal repair request: bypassing memory side effects
[MemoryProxy] Extraction already in-progress; marked pending catch-up
[MemoryProxy] Pending extraction detected; starting catch-up
[MemoryProxy] Catch-up extraction complete: X facts, Y events
```

不应该再看到 repair 请求像普通用户对话一样触发：

```text
Chat request intercepted — stream: false
Session: ...
Continuity: ...
Extraction already in-progress...
```

最终目标：

```text
repair 不污染 memory
extraction 不无痕 skip
pending 能追赶最新消息
catch-up 不递归
长期记忆延迟存在但不会无限扩大
```
